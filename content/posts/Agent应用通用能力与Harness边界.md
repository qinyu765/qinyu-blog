最近看 Agent 应用时，我越来越觉得最容易混乱的地方不在模型本身。真正让人分不清边界的，是一堆词同时涌进来：上下文、记忆、工具、规划、多 Agent、评估、权限、运行时，最后再加一个 Harness。每个词都像是 Agent 的一部分，但它们承担的工程职责并不相同。

如果要做一次系统调研，最好先把问题拆开。哪些能力几乎所有 Agent 应用都会遇到，哪些能力只在特定类型里出现，哪些又属于把 Agent 跑起来、管起来、测起来的外层设施。这个分层一旦清楚，后面再看框架、论文或产品，就不会只是在堆名词。

## 先把 Agent 拆成一条执行链

一个 Agent 应用可以先粗略理解成一条循环链路。用户给出目标，系统收集上下文，模型做出判断，必要时调用工具，然后根据工具结果继续推理或结束任务。这个过程看起来像一次聊天，工程上却更接近一个带状态的任务执行系统。

这条链路里，LLM 只是决策核心。它负责理解目标、选择下一步动作、组织最终回答，但它拿到什么信息、能调用什么工具、失败后怎么恢复，通常由外层系统决定。很多 Agent 的差异，恰恰来自这些外层系统的设计。

## 通用能力：每个 Agent 都绕不开的底座

几乎所有 Agent 应用都会遇到同一组基础问题。上下文管理决定模型在当前回合能看到什么，记忆系统决定跨会话能保留什么，工具系统决定 Agent 能对外部世界做什么，规划与执行机制决定它如何把目标变成动作序列。

这些能力可以放进一张工程地图里看。它们不是按代码目录天然分开的模块，更像是一组互相牵制的设计面：上下文太少会误判，记忆太多会污染判断，工具太强会带来权限风险，规划太复杂又会增加成本和延迟。

| 能力层 | 主要问题 | 常见设计点 |
|---|---|---|
| 上下文管理 | 当前任务需要哪些信息 | 对话历史、文件片段、网页内容、检索结果、上下文压缩 |
| 记忆系统 | 哪些信息应该跨任务保留 | 用户偏好、项目知识、历史决策、记忆写入与遗忘 |
| 工具系统 | Agent 能采取哪些动作 | 工具描述、参数生成、调用权限、结果解析、失败恢复 |
| 规划执行 | 如何把目标变成步骤 | ReAct、Plan-and-Execute、工作流、重试、暂停与恢复 |
| 状态管理 | 任务执行到哪里了 | 子任务状态、工具状态、用户确认状态、错误状态 |
| 安全评估 | 如何控制风险并衡量效果 | 沙箱、审批、日志、成本、成功率、幻觉率 |

把这些层放在一起看，会发现 Agent 应用的工程难点并不只在“让模型更聪明”。很多时候，真正的问题是让模型在有限信息里做出稳定动作，并且让每一步都能被观察、限制和复盘。

## Harness：把能力装进可运行环境

Harness 这个词容易和 Agent 框架混在一起。为了调研方便，可以把 Harness 理解成“运行脚手架”或“执行外壳”：它负责把模型、提示词、工具、日志、评估、权限和测试环境连接起来，让 Agent 能在一个可控的场景里运行。Agent 负责决策，Harness 负责让决策发生在受控边界内，并留下足够的证据。

在代码 Agent 里，Harness 可能是一套工作区沙箱、文件读写权限、测试命令、Git 状态检查和审批机制。在浏览器 Agent 里，Harness 可能包含浏览器实例、页面状态、截图、DOM 操作接口和导航约束。在评测场景里，Harness 还会提供任务集、标准答案、打分器、日志追踪和复现实验的入口。

这样区分之后，很多概念会清楚很多。上下文、记忆、工具和规划属于 Agent 能力层；Harness 属于承载和约束这些能力的运行层。前者回答“Agent 怎么想、怎么做”，后者回答“它在哪里做、按什么规则做、做完如何验证”。

## 不同 Agent 的差异来自工作现场

通用能力只是底座，真正的差异来自 Agent 所在的工作现场。代码 Agent 面对的是仓库、测试、构建和版本控制，所以它必须理解文件结构、局部修改、回归验证和提交边界。浏览器 Agent 面对的是页面状态和交互流程，所以它更依赖截图、DOM、点击、输入和等待策略。

研究型 Agent 的关键能力在信息来源处理。它要搜索、筛选、引用、核查，还要区分资料的时效性和可靠性。客服 Agent 更关心知识库命中、用户情绪、工单流转和人工接管。数据分析 Agent 则围绕表格、SQL、统计、图表和解释展开。

多 Agent 系统的差异更明显。它关心的已经不只是一个 Agent 如何完成任务，还包括任务如何分派、角色如何协作、意见冲突如何解决、共享状态如何维护。只要进入多 Agent 场景，通信协议和协调机制就会变成一等公民。

## 多 Agent：协作带来的收益和代价

多 Agent 很容易让人兴奋，因为它看起来像是在复制人类团队：规划者负责拆任务，执行者负责操作，审查者负责检查，研究者负责找资料。这个模式在复杂任务里确实有价值，尤其当子任务可以并行、角色边界清楚、每个 Agent 都有明确输入输出时。

但多 Agent 也会快速放大系统复杂度。单 Agent 出错时，问题通常集中在上下文、工具或提示词；多 Agent 出错时，还要排查消息传递、状态同步、角色重叠、重复劳动和相互误导。调研时可以把多 Agent 当成一种组织形态，而不是默认升级方向。

更稳妥的判断方式是先看任务结构。如果任务天然可以拆成互不干扰的子问题，多 Agent 能带来并行和专业化收益。如果任务需要频繁共享细节、不断互相等待，单 Agent 加清晰工具链往往更可控。

## 调研时可以抓住的主线

我会把 Agent 应用调研分成三层。第一层是能力层，重点看上下文、记忆、工具、规划、状态、安全和评估。第二层是运行层，重点看 Harness、沙箱、权限、日志、任务恢复和部署方式。第三层是领域层，重点看代码、浏览器、研究、客服、数据分析、多 Agent 这些具体场景里新增了什么能力。

这个框架的好处是能避免把所有概念放在同一个篮子里。比如“工具调用”是通用能力，“浏览器点击”是浏览器 Agent 的领域工具，“Playwright 环境和截图验证”则更接近 Harness。三者都和行动有关，但调研时应该放在不同位置分析。

继续往下做，可以按一个固定问题去看每个 Agent 产品或框架：它怎么管理上下文，怎么保存记忆，怎么接工具，怎么规划，怎么处理中断，怎么做权限控制，怎么评估结果，Harness 提供了哪些运行保障。问完这一组问题，大多数 Agent 应用的轮廓都会显现出来。

所以，Agent 调研的重点不是收集更多名词，而是建立一张可复用的工程地图。地图上既有通用底座，也有领域差异，还有 Harness 这类把能力落地的运行设施。只要这三层分清，后面无论看代码 Agent、浏览器 Agent，还是多 Agent 协作系统，都能更快判断它真正解决了什么问题。
