## 1. 什么是 React 类组件

在 React 早期版本中，组件主要通过 ES6 `class` 定义。

### 1.1 基本结构

```jsx
class Counter extends React.Component {
  constructor(props) {
    super(props);
    this.state = { count: 0 };
  }

  componentDidMount() {}

  render() {
    return <div>{this.state.count}</div>;
  }
}
```

### 1.2 类组件的核心机制

| 能力   | 实现方式                |
| ---- | ------------------- |
| 状态管理 | this.state          |
| 更新状态 | this.setState       |
| 生命周期 | componentDidMount 等 |
| 逻辑复用 | HOC / render props  |
| 组件模型 | 面向对象实例              |

本质上：

> 类组件 = 面向对象的 UI 生命周期容器

---

## 2. 为什么现在不推荐类组件

类组件没有被废弃，但属于**遗留范式（Legacy API）**。

### 2.1 逻辑复用困难

类组件依赖：

* 高阶组件
* render props
* 继承

导致：

* 组件嵌套复杂
* 逻辑与结构耦合
* 类型系统难推导

Hooks 的函数组合模型更直接。

---

### 2.2 生命周期模型复杂

类组件：

```
componentDidMount
componentDidUpdate
componentWillUnmount
```

Hooks：

```js
useEffect(() => {
  return () => {};
}, []);
```

变化本质：

> 从生命周期驱动 → 副作用驱动

---

### 2.3 this 与实例语义问题

类组件常见问题：

* this 绑定
* 实例状态可变
* 行为与生命周期耦合

函数组件：

* 无实例
* 无 this
* 渲染纯函数化

---

### 2.4 新能力围绕函数组件演进

| 新特性               | 类组件支持  |
| ----------------- | ------ |
| Hooks             | ❌      |
| 并发渲染              | 兼容但非核心 |
| Server Components | ❌      |

结论：

> 类组件仍可用，但不再是未来方向。

---

## 3. React 是否从 OOP 过渡到 FP？

不是范式替代，而是组件抽象方式改变。

### 3.1 早期模型

```
组件 = 对象实例
状态 = 对象属性
生命周期 = 对象方法
```

### 3.2 当前模型

```
组件 = 状态计算函数
UI = f(state)
副作用 = 声明式描述
```

本质变化：

> 从“组件是对象” → “组件是计算过程”

---

## 4. Hooks 不是类组件的封装

常见误解：

❌ Hooks 是类组件语法糖
✔ Hooks 是新的组件执行模型

### 4.1 状态存储位置变化

类组件：

```
实例对象
  └── state
```

函数组件：

```
React 运行时（Fiber）
  └── Hook 状态链表
```

关键区别：

| 类组件    | 函数组件    |
| ------ | ------- |
| 状态属于实例 | 状态属于运行时 |
| 生命周期方法 | 副作用声明   |
| 面向对象模型 | 函数执行模型  |

---

## 5. React 没有抛弃面向对象

准确理解：

> React 在组件层采用函数组合
> 在运行时实现仍使用对象系统

这是一种**分层架构设计**。

---

## 6. React 架构的三层结构

### 6.1 开发者编程模型（函数式）

```js
function Component() {
  const [state] = useState();
  return UI;
}
```

特征：

* 无实例
* 函数组合
* 声明式副作用

---

### 6.2 React 运行时（对象系统）

内部维护：

```
FiberNode {
  memoizedState
  updateQueue
  child
  sibling
}
```

职责：

* 状态存储
* 更新调度
* 副作用管理
* 渲染执行

这是典型工程运行时系统。

---

### 6.3 宿主环境（浏览器或其他平台）

* DOM API
* 事件系统
* 渲染引擎

React 运行在此环境之上。

---

## 7. Vue 与 React 的共同趋势

Vue 也经历相同演进：

| 旧模型         | 新模型             |
| ----------- | --------------- |
| Options API | Composition API |

共同目标：

> 状态逻辑函数化
> 行为可组合
> 副作用声明化

行业趋势：

> 编程模型函数式化
> 运行时工程系统化

---

## 8. 什么是运行时（Runtime）

严格定义：

> 运行时是程序执行期间提供环境、状态管理与执行机制的系统。

运行时的基本职责：

| 职责   | 说明   |
| ---- | ---- |
| 执行代码 | 运行程序 |
| 管理状态 | 存储数据 |
| 调度执行 | 控制时序 |
| 提供语义 | 定义行为 |

运行时关注的是：

👉 程序如何运行
而不是如何编写

---

## 9. 运行时的层级划分

### 9.1 JavaScript 语言运行时

代表：

* Node.js
* Bun

提供：

* JS 执行引擎
* 事件循环
* 系统 API
* 内存管理

这是**语言级运行时**。

---

### 9.2 框架运行时（应用级）

React / Vue 运行时提供：

* 组件系统
* 状态管理
* 更新调度
* 副作用语义

这是**框架级运行时**。

---

### 9.3 运行时分层模型

```
JavaScript 运行时（Node / Bun / 浏览器）
        ↓
UI 框架运行时（React / Vue）
        ↓
应用代码
```

关系：

| 层级    | 作用       |
| ----- | -------- |
| 语言运行时 | 让 JS 能执行 |
| 框架运行时 | 让组件能工作   |
| 应用代码  | 实现业务     |

---

## 10. 为什么现代框架采用这种分层

### 10.1 对开发者

需要：

* 简洁表达
* 可组合逻辑
* 可预测行为

函数式接口最合适。

---

### 10.2 对框架实现

需要：

* 调度能力
* 性能优化
* 内存管理
* 状态控制

这些属于系统工程问题，更适合对象模型。

---

## 11. 最终统一认知模型

可以用一句话总结整套知识体系：

> 现代前端框架在编程模型上函数式化，在运行时实现上工程系统化。

或者更具体：

> React 与 Vue 在使用层是函数式组合系统，在内部实现层是对象化运行时系统。

这不是范式替换，而是：

👉 抽象层重构
👉 职责分层
👉 表达模型升级

---

## 12. 核心知识点速记

### React 架构定位

* 类组件：历史主流
* 函数组件：当前标准
* Hooks：执行模型
* Fiber：运行时对象系统

### 编程范式关系

* 不存在 OOP → FP 替代
* 存在接口层函数式化
* OOP 下沉到框架内部

### 运行时概念

* Node/Bun：语言运行时
* React/Vue：框架运行时
* 运行时可以分层存在
