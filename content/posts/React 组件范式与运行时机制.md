如果你是在 Hooks 时代入行的前端开发者，可能从来没写过类组件。`useState`、`useEffect` 这些 API 就是你认识 React 的方式，类组件只存在于老项目的代码和面试题里。但如果你想真正理解 React 的设计哲学——为什么状态不存在组件里？为什么副作用要用声明式描述？为什么 React 团队要做这么大的范式转换？——就需要回到类组件的时代，看看这条演进路径到底经历了什么。

## 类组件：面向对象的 UI 容器

React 早期的组件模型是纯粹的面向对象设计。组件就是 ES6 class 的实例，状态存在 `this.state` 里，更新通过 `this.setState` 触发，而 `componentDidMount`、`componentDidUpdate`、`componentWillUnmount` 这些生命周期方法则定义了实例在不同阶段的行为。本质上，类组件就是一个"面向对象的 UI 生命周期容器"。

这套模型在语义上是清晰的，但在实际使用中暴露了几个结构性问题。逻辑复用极度依赖高阶组件和 render props，组件嵌套层级越来越深，逻辑和结构严重耦合。`this` 的绑定问题在异步回调和事件处理中反复制造 bug。多个不相关的逻辑被塞进同一个生命周期方法里（比如在 `componentDidMount` 里同时做数据请求、事件监听和定时器设置），代码的可读性和可维护性随着功能增长急剧下降。

更关键的是，类组件的实例模型和 React 后续的演进方向产生了根本性冲突。并发渲染、Server Components 这些新能力都围绕函数组件设计，类组件在兼容层面越来越边缘化。React 官方文档已经把类组件标记为 Legacy API——仍然可用，但不再是未来方向。

## 函数组件与 Hooks：不是语法糖，是新的执行模型

一个常见的误解是把 Hooks 当成类组件的语法糖。事实上，Hooks 代表了一套全新的组件执行模型，两者在状态存储位置、生命周期概念、逻辑组织方式上都有根本区别。

在类组件中，状态属于组件实例——每个组件 `new` 出来一个对象，`this.state` 就是这个对象的属性。在函数组件中，状态不属于组件本身，而是存储在 React 运行时的 Fiber 节点上，通过 Hook 状态链表进行管理。函数组件每次渲染都是一次函数调用，没有实例、没有 `this`、没有可变的内部状态。所有的"记忆"能力都来自运行时，而非组件自身。

生命周期的概念也被彻底重构。类组件围绕"组件的生命阶段"来组织逻辑——挂载时做什么、更新时做什么、卸载时做什么。函数组件用 `useEffect` 把这些阶段收敛成了"副作用声明"——你描述的是"当这些依赖变化时，执行这段副作用"，而不是"在组件的第几个生命阶段执行"。这是从生命周期驱动到副作用驱动的转变，思维模型更接近数据流而非对象行为。

## 分层架构：编程模型与运行时的分离

理解了函数组件的执行模型之后，一个自然的问题是：如果函数组件没有实例、没有 `this`、状态也不在自己身上，那这些东西到底在哪里？答案指向 React 架构中一个精妙的分层设计。

React 的架构可以分为三层。最上层是开发者的编程模型——你写的函数组件、`useState`、`useEffect`，这一层是纯函数式的，无实例、声明式、可组合。中间层是 React 的运行时系统（Fiber 架构），它维护着一棵 FiberNode 组成的树，每个节点上挂载着 `memoizedState`（Hook 链表）、`updateQueue`（更新队列）、`child`/`sibling`（树结构关系）等属性。这一层是典型的面向对象的工程运行时系统，负责状态存储、更新调度、副作用管理和渲染执行。最底层是宿主环境——浏览器的 DOM API、事件系统和渲染引擎。

所以 React 并没有"抛弃面向对象"。更准确的说法是：它在组件层（开发者接触的部分）采用了函数式组合，在运行时实现层（开发者不直接接触的部分）继续使用对象系统。这种分层是有意为之的——开发者需要简洁的表达、可组合的逻辑和可预测的行为，函数式接口最合适；运行时需要调度能力、性能优化、内存管理和状态控制，这些是系统工程问题，对象模型更胜任。

## 行业共识：不止 React 在这么做

这种"编程模型函数式化、运行时工程系统化"的分层并非 React 独有。Vue 经历了几乎相同的演进：从 Options API（本质上是配置对象驱动）到 Composition API（函数组合驱动），核心目标完全一致——让状态逻辑可组合、行为可复用、副作用可声明。

两个主流框架同时走向同一个方向，说明这不是某个团队的风格偏好，而是行业在工程实践中得出的共识。当应用复杂度增长到一定程度，面向对象的组件模型在逻辑复用和状态管理上的局限性会系统性地暴露出来，而函数式组合提供了更扁平、更可预测的组织方式。

把这条脉络看清楚之后，很多日常开发中的细节就有了解释。为什么 Hooks 的调用顺序不能变？因为状态链表依赖顺序索引。为什么 `useEffect` 的依赖数组那么重要？因为运行时靠它来判断副作用是否需要重新执行。为什么 React 能做并发渲染？因为 Fiber 架构把渲染过程变成了可中断的工作单元。这些设计决策不是孤立的 API 设计，而是整套分层架构的必然产物。
