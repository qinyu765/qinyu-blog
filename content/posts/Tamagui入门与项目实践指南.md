在 React Native 生态中，样式方案一直是一个令人纠结的话题。`StyleSheet.create` 功能受限，Styled Components 在跨平台场景下需要大量适配，而且无论哪种方案，运行时的样式计算都是性能隐患。Tamagui 尝试从一个不同的角度解决这个问题：如果样式能在构建时就确定下来，运行时根本不需要计算，性能问题是不是就不存在了？

## 编译时优化：Tamagui 的核心竞争力

Tamagui 最与众不同的地方不是它的组件库，而是它的优化编译器。在构建阶段，编译器会分析组件的样式 props，把能够静态确定的样式提取为原子 CSS（Web 端）或预计算的样式对象（Native 端）。这意味着运行时几乎不需要做样式解析和合并的工作，性能开销被前置到了构建过程中。

这和传统的 CSS-in-JS 方案形成了鲜明对比。Styled Components 和 Emotion 在运行时动态生成样式，每次组件渲染都可能触发样式计算。Tamagui 则尽可能在编译时完成这些工作，运行时只负责应用已经计算好的结果。对于列表页、瀑布流这类频繁渲染的场景，这种差异会直接体现在帧率和交互流畅度上。

当然，编译时优化有一个前提：样式 props 需要是静态可分析的。如果你传入的是一个运行时才能确定的变量，编译器无法提前处理，就会退回到运行时计算。所以在使用 Tamagui 时，尽量用 token 引用和静态值来获得最大的优化收益。

## 设计令牌：用系统化的方式管理设计决策

Tamagui 内置了完整的设计令牌（Design Tokens）系统，这是它作为"设计系统框架"而非"样式工具"的关键特征。Tokens 本质上是可复用的设计值——颜色、尺寸、间距、圆角、层级——通过 `$` 前缀在组件 props 中引用。

颜色系统采用 1-12 的色阶设计，每个色系（蓝、灰、红、绿等）都有从最浅到最深的 12 个梯度。这套色阶有明确的使用约定：1-2 用作背景色，3-4 用作交互元素的背景，5-6 用于边框和分割线，9 是主色调（按钮、链接），10-11 分别对应 hover 和 pressed 状态，12 用于文字。这种规约让团队不需要反复讨论"按钮应该用什么蓝"，直接引用 token 就能保证视觉一致性。

主题系统建立在 token 之上，支持嵌套切换。你可以在全局设定 light 主题，然后在某个区域嵌套一个 dark 主题，内部组件自动适配。token 的值会随主题切换而变化，组件代码不需要做任何修改——这是声明式设计系统的核心优势。

## 简写语法与布局组件

Tamagui 提供了一套可配置的简写属性系统。`w` 代替 `width`，`bg` 代替 `backgroundColor`，`ai` 代替 `alignItems`，`jc` 代替 `justifyContent`。这些简写在 `tamagui.config.ts` 中声明，项目可以自由扩展。简写不是必须的——Tamagui 默认允许简写和完整属性名同时使用——但在日常开发中确实能显著减少代码量。

布局方面，Tamagui 提供了语义化的布局组件：`XStack` 默认 `flexDirection: 'row'`（水平排列），`YStack` 默认 `flexDirection: 'column'`（垂直排列）。相比直接使用 `View` 然后手动指定 flex 方向，语义化组件让布局意图一目了然。配合 `gap` 属性和简写语法，一个典型的卡片布局只需要几个 props 就能表达清楚。

## 在第三方组件中使用 Token

Token 引用（如 `$blue9`）只在 Tamagui 自身的组件 props 中有效。当你需要把颜色值传给第三方组件（SVG 图标库、图表库、动画库等）时，它们不认识 `$` 前缀，需要先获取 token 的实际值。Tamagui 提供了 `useTheme` Hook 来解决这个问题：

```tsx
import { useTheme } from 'tamagui';

const theme = useTheme();
const primaryColor = theme['$blue9']?.val;  // 获取实际颜色值
```

`useTheme` 返回的对象中，每个 token 都包含 `val`（实际值）和 `variable`（CSS 变量名，仅在 Web 端有效）两个属性。通过动态构建 token 名（比如用模板字符串拼接色系和色阶），还可以实现主题色的运行时切换。

这个设计也说明了 Tamagui 的一个架构选择：token 系统是封闭的，只在 Tamagui 组件体系内自动生效，和外部世界交互时需要显式地"解包"。这保证了编译器能对 Tamagui 组件做深度优化，代价是和第三方库集成时需要多一步手动处理。

## 性能优化的实践原则

Tamagui 项目中的样式优先级有一个明确的层级：最优先使用 Tamagui 组件 props（能被编译器优化），其次是 `StyleSheet.create`（原生方案的标准做法），而内联 `style` 属性应该尽量避免，因为它会跳过编译器优化且每次渲染都创建新对象。

在组件层面，`memo` 包装和 `useMemo` 缓存样式计算是标准做法，和 React 项目中的性能优化一脉相承。使用 token 引用而非硬编码颜色值，除了维护性的好处之外，还让编译器有更大的优化空间。

Tamagui 代表了跨平台 UI 开发的一个趋势：不再只是"让代码跑在多端"，而是在多端一致性的基础上，把性能优化做到编译时，把设计决策做到系统化。对于需要同时覆盖 Web 和 Native、且对性能和设计一致性有要求的项目，它提供了一个比 StyleSheet + 手动适配更完整的解决方案。
