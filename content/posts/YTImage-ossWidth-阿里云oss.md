移动端应用中，图片是最常见的性能瓶颈之一。一个电商列表页可能有几十张商品缩略图，每张图片的大小直接影响页面加载速度和内存占用。但很多开发者在写代码时只关注图片的显示尺寸——Figma 上标注的是 120px 就写 120px——而忽略了一个关键问题：从服务器下载的图片原图可能是 1000px 甚至更大。一张显示区域只有 120px 的图片，下载了 1000px 的原图，90% 以上的像素都在传输和解码后被丢弃了。

## 为什么不能简单地"按显示尺寸请求"

这个问题的答案藏在设备像素密度里。现代移动设备的屏幕不是 1:1 的——一个逻辑像素可能对应 2 个甚至 3 个物理像素。iPhone 标准版的像素密度是 2x（一个逻辑像素对应 2x2=4 个物理像素），Pro 系列是 3x（一个逻辑像素对应 3x3=9 个物理像素）。

这意味着，如果你在 2x 屏幕上显示一个 120pt 的头像，实际需要的图片尺寸是 240px，才能让每个物理像素都有对应的图片像素，避免模糊。如果只请求 120px 的图片，系统会把它拉伸到 240px 来填满物理像素，结果就是肉眼可见的模糊。反过来，如果直接下载 1000px 的原图，清晰度确实够了，但白白浪费了大量带宽和内存。

所以最优策略是：根据显示尺寸和设备像素密度，计算出"刚好够用"的物理像素宽度，然后只请求这个尺寸的图片。

## 服务端图片处理：让 CDN 帮你裁剪

这种"按需裁剪"的能力通常由 CDN 或对象存储服务提供。以阿里云 OSS 为例，你可以在图片 URL 后面追加处理参数来控制返回的图片尺寸：

```
https://cdn.example.com/avatar.jpg?x-oss-process=image/resize,w_240
```

这条 URL 告诉 OSS："我只需要宽度为 240px 的版本。"OSS 会在服务端用高质量插值算法把原图缩放到指定宽度，保持宽高比，然后返回一个体积小得多的图片。原图可能是 800KB，缩放后可能只有 40KB，流量节省超过 95%。

这个处理过程发生在服务端，不消耗客户端的计算资源。而且 CDN 通常会缓存处理后的结果，同一张图片的同一个尺寸参数不会重复计算。

## 组件层面的设计模式

在实际的 React Native 项目中，这套逻辑通常被封装在图片组件内部。开发者只需要指定两个参数：图片在屏幕上的显示尺寸，以及期望的逻辑像素宽度。组件内部会自动获取设备的像素密度（通过 `PixelRatio.get()`），计算出物理像素宽度，然后拼接到 URL 参数里。

```typescript
// 核心逻辑的伪代码
const pixelRatio = Math.min(PixelRatio.get(), 3); // 限制最大 3x
const physicalWidth = Math.round(logicalWidth * pixelRatio);
const processedUrl = `${originalUrl}?x-oss-process=image/resize,w_${physicalWidth}`;
```

一个好的图片组件还会实现"首次锁定"策略：组件初始化时确定请求尺寸后就不再改变，即使外部 props 变化也不会触发重新请求。这是因为图片 URL 的变化会导致缓存失效，产生不必要的网络请求。对于动态尺寸的场景，更稳妥的做法是用不同的组件实例而非改变同一个实例的尺寸参数。

对于本地资源图片（通过 `require()` 导入的），不需要也不能做服务端处理——本地图片不经过网络，它就是什么尺寸就是什么尺寸。代码中通常通过检查 URL 是否以 `http://` 或 `https://` 开头来区分网络图片和本地图片，只对网络图片追加处理参数。

## 实际优化效果

以一个 Figma 设计稿中 120px 宽度的用户头像为例：在 2x 屏幕上，不做优化直接下载 375px 的原图约为 150KB，做了按需裁剪后下载 240px 版本约为 60KB，节省 60% 流量。在 4G 网络下，单张图片的加载时间从约 600ms 降到 240ms。这个差距在列表场景中会被放大——一个包含 20 张缩略图的列表页，优化前后的总流量和加载时间差距可以达到数倍。

阿里云 OSS 的图片处理能力远不止宽度缩放，还支持裁剪、质量压缩、格式转换（比如转 WebP）等操作。在移动端项目中，通常只用到宽度缩放就能覆盖绝大多数场景。但如果项目对图片性能有极致要求，结合格式转换（WebP 在同等画质下体积比 JPEG 小 25-35%）和质量参数的调整，还有进一步优化的空间。

图片优化看起来是一个很"小"的点，但在移动端应用中，它往往是用户感知最直接的性能瓶颈。理解设备像素密度、服务端处理、客户端缓存这几个环节的配合关系，把"按需请求"的思维内化到日常开发中，就能在不增加开发复杂度的情况下，持续享受这条优化路径带来的收益。
