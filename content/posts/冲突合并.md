第一次遇到 Git 冲突的时候，终端里那一串 `CONFLICT (content): Merge conflict in...` 让我心跳加速。文件里突然冒出来的 `<<<<<<<`、`=======`、`>>>>>>>` 像是某种错误信号，让人下意识觉得自己做错了什么。但冲突不是错误，它是 Git 在告诉你："这个地方两边都改了，我不知道该听谁的，你来决定。"理解这一点之后，处理冲突就从一件令人焦虑的事变成了一个可以建立肌肉记忆的标准流程。

## 冲突为什么会发生

Git 冲突的本质是同一位置的内容在两个分支上被不同方式修改，Git 无法自动判断哪个版本是正确的。最常见的场景是两个人改了同一个文件的同一段代码：开发 A 在 `feature-A` 改了某个函数的实现，开发 B 在 `feature-B` 也改了同一个函数，当某一方合并时就会冲突。

除了代码层面的冲突，还有一些更隐蔽的情况。比如一个人删除了某个文件，另一个人在自己的分支上修改了这个文件——合并时 Git 不知道这个文件该存在还是该消失。又比如同一个文件被不同方式重命名或移动，Git 同样无法自动合并。还有一种结构性原因：分支长期不跟进主干更新，变更大量堆积，最终一次合并时会爆发大面积冲突。

## 冲突出现时的标准处理流程

当你执行 `git merge` 或 `git rebase` 遇到冲突时，Git 会中断操作并标记冲突文件。用 `git status` 可以看到哪些文件处于 `both modified` 状态。打开这些文件，你会看到冲突标记：`<<<<<<< HEAD` 和 `>>>>>>> branchB` 之间分别是两个分支的版本，用 `=======` 隔开。

处理方式有三种选择。最简单的是完全采用某一边的版本，适合配置文件或 lock 文件这类不需要合并逻辑的场景，用 `git checkout --ours` 或 `git checkout --theirs` 就能搞定。最常见的是手工合并，需要你理解双方的变更意图，把两边新增的逻辑合在一起，注意变量重命名、新增参数、已删除代码等细节。第三种情况是冲突位置涉及你不熟悉的代码，这时候最稳妥的做法是找相关开发确认，不要凭感觉乱合。

编辑完成后，删除所有冲突标记，用 `git add` 暂存已解决的文件，然后 `git commit` 完成合并。如果是在 rebase 过程中，则用 `git rebase --continue` 继续下一个 commit 的应用。rebase 和 merge 在冲突处理上的核心区别是：merge 一次性解决所有冲突并生成一个合并节点，rebase 可能需要逐个 commit 解决冲突但最终形成线性历史。

## 减少冲突的工程实践

冲突无法完全避免，但可以通过工程实践大幅降低频率和严重程度。最有效的策略是缩短分支生命周期——每个分支只做一件事，1 到 3 天内合并回主干。分支存在的时间越长，和主干偏离越大，冲突的概率和复杂度都会指数上升。

与之配合的是频繁同步主干。在 feature 分支上定期执行 `git fetch` 加 `git merge origin/main`（或 `git rebase origin/main`），把主干的最新变更拉进来，让冲突早发现、早解决，避免最终合并时一次性爆发。

对于高频编辑的文件（路由表、接口定义、公共常量），可以通过拆分文件、模块化组织来降低多人同时修改同一文件的概率。另外，统一代码格式化工具（Prettier、ESLint 等）也很重要——很多表面上的"冲突"其实只是格式差异，统一格式后这类无意义的 diff 就消失了。

## 团队协作中的合并规范

在团队层面，合并方向和责任归属需要明确约定。通常的规则是功能分支合向中心分支（develop 或 main），避免分支之间互相 merge 形成复杂的历史网络。谁发起 PR，谁负责解决冲突——涉及复杂业务逻辑时可以和相关模块负责人一起看，但主责任在 PR 发起者。

GitHub 提供了三种合并策略：Merge commit 保留合并节点，适合需要审计追踪的场景；Squash and merge 把一堆小 commit 压缩成一个，适合功能分支的日常合并；Rebase and merge 形成线性历史，适合追求干净提交记录的团队。选择哪种取决于团队的审计需求和历史管理偏好。

关于提交信息，纯粹的冲突解决提交建议用 `chore: resolve merge conflicts` 这样的格式。如果在解决冲突的过程中顺带修了 bug 或改了功能，最好拆成两个提交：一个专门做冲突解决，一个做功能修改。这样在排查问题时能清晰地分辨哪些变更是有意为之，哪些只是合并的副产物。

冲突处理说到底是一个关于沟通和规范的问题。技术上它并不复杂——标记清楚、选择明确、验证通过就行。真正的挑战在于团队是否有一致的分支策略、合并规范和协作习惯。把这些"软"的部分理顺了，冲突就真的只是日常开发中一个不值得紧张的小环节。
