`export default` 和 `export { ... }` 是 JavaScript 模块系统里最基础的两种导出方式。大多数开发者在写第一个组件的时候就接触了它们，但很少有人在项目做大之后回过头来想：这两种方式在工程化层面到底有什么不同？什么时候该用哪个？这些问题看似简单，但答案关系到代码的可维护性、重构安全性、以及团队协作的效率。

## 语法差异背后的核心分歧

从语法层面看，区别很直观。默认导出一个模块最多只能有一个，导入时名字可以随便起：`import AnyName from './mod'`。命名导出可以有多个，导入时必须使用同名（或用 `as` 显式改名）：`import { foo } from './mod'`。

但真正的分歧不在语法，而在"名字是否被模块强约束"这个问题上。默认导出的导入名不受模块约束——同一个模块在不同文件里可能被叫不同的名字，全局搜索和代码审查变得困难。命名导出则相反，导出名就是契约，所有导入方必须遵守，跨文件的一致性天然更强。

这个差异在小项目中几乎感受不到。但当项目有几十个文件互相引用，团队有多人协作时，默认导出的"自由命名"反而变成了负担。把 `ConfirmExchange` 导入成 `ConfirmModal`，TypeScript 不会报错——因为默认导出的名字本来就不受约束。而命名导出在重命名时会立即触发编译错误，强迫你系统性地修复所有引用点。

## 工程化视角：Tree-shaking 与互操作

在打包优化层面，命名导出通常更利于 tree-shaking。现代打包器对 ESM 做静态分析时，命名导出的"按需引用"意图更加明确。但需要注意的是，能否真正 tree-shake 还取决于是否存在副作用、构建配置等多个因素，并非"命名导出就一定更小"。

另一个容易踩坑的地方是和 CommonJS 的互操作。当 ESM 和 CJS 混用时，`default` 的行为会变得微妙，`esModuleInterop`、`allowSyntheticDefaultImports` 等 TypeScript 配置都会影响导入方式。如果项目全程使用 ESM（现代前端项目通常如此），只需要记住一件事：ESM 里 `default` 和 `named` 是两套独立的导出机制，不要混淆。

## 实际项目中的选择策略

在 React 和 React Native 项目中，选择策略可以按文件类型来定。路由和页面文件适合使用默认导出——像 Expo Router 这类文件即路由的框架，本身就要求页面组件以 `export default` 导出，而且"一个文件一个页面"的语义也天然适合默认导出。

通用组件、工具函数、Hooks、常量这类文件则更适合命名导出。原因很实际：这类文件往往会随着项目演进逐步增加导出内容，一开始只有一个 `formatPrice`，后来加上 `formatDate`、`formatCurrency`，命名导出在扩展性上更好。而且 IDE 对命名导出的自动导入支持更准确，不会因为名字不一致导致补全失败。

重新导出（barrel 文件）时也有一个细节值得注意：`export * from './x'` 不会把默认导出一起导出。如果需要转发默认导出，必须显式写 `export { default as X } from './x'`。

## 一套可落地的团队规范

如果团队需要统一规则，以下是一套经过验证的实用方案。路由和页面文件使用 `export default`；工具函数、Hooks、常量、领域模型使用命名导出；组件文件看约定——如果团队坚持"一个文件一个组件"，默认导出可以，但如果组件文件会附带子组件、变体、类型导出，命名导出更合适。

有一条通用原则：避免在一个文件里同时使用大量的默认导出和命名导出。虽然语法上完全合法，但会让使用者困惑"到底该默认导入还是解构导入"。保持每个文件的导出方式一致，认知负担就会低很多。

选择默认导出还是命名导出，归根结底不是一个语法偏好问题，而是一个关于"代码如何在团队中流动"的工程决策。在小项目中两者差距微乎其微，但在多人协作的中大型项目中，命名导出在重构安全性和一致性上的优势会随着项目规模线性放大。
